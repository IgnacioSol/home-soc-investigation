What Is This Project

This project simulates a real workday for a SOC Tier 2 analyst: receiving an escalated alert, investigating a multi-stage intrusion from initial execution through lateral movement, building a complete attack timeline, and delivering a professional incident report.

The investigation runs against real attack simulation logs from Splunk's own Threat Research Team, using the same tooling (Splunk, SPL queries) used in enterprise Security Operations Centers.

Incident Scenario

A Windows endpoint at AcmeCorp triggered multiple security alerts. Investigation reveals a full APT-style intrusion: malicious PowerShell execution, LSASS credential dumping via Mimikatz, registry-based persistence, and RDP lateral movement to an internal host. The analyst role is SOC Tier 2 — escalated case, full investigation required.

Key Detection Logic — Malicious PowerShell (T1059.001)

One of the core detections in this investigation targets malicious PowerShell execution. The Sigma rule converts to SPL that looks for four specific flags that together indicate an attacker, not a legitimate user:

-WindowStyle Hidden
Hides the PowerShell window from the screen. A legitimate user never needs to hide their own terminal.
-ExecutionPolicy Bypass
Windows blocks unsigned scripts by default. This flag bypasses that policy — a signal that the script would otherwise be rejected by the system.
-NoProfile
Skips loading the user profile, avoiding activity logging that would normally occur on profile load.
-enc
Runs a Base64-encoded command — the most common technique to obfuscate malicious PowerShell so it can't be read at a glance.

All four together = someone running PowerShell who doesn't want to be seen or traced. This is what the Sigma rule detects and sigma-cli translates to an SPL query runnable in Splunk.

Connection to Project #1

The detection rules used in this investigation were written from scratch in Project #1 (sigma-detection-rules), covering each ATT&CK technique in the Sigma format — a vendor-neutral standard used by real threat detection teams. This project is where those rules are converted to SPL and validated against real attack data, completing the full detection engineering lifecycle: write rule → translate → test against real logs → investigate → report.
